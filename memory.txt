- That pointed to the web server being starved by blocking BLE request/response handling in the earlier implementation.
- The blocking pattern that needed changing was the old requestBatterySnapshot()-style logic: send request, wait in a loop for reply/timeout, then move on.
- We decided to refactor BLE polling into a non-blocking per-battery state machine.
- Requested BLE polling changes:
  - per-battery request state
  - fields like requestInFlight, lastRequestMs, requestDeadlineMs
  - request fresh data every 2000 ms per connected battery
  - let notify callback complete in-flight request asynchronously
  - mark timeout/failure without blocking the loop
  - keep persistent classic BLE connections
  - preserve aggregate CAN output and multi-battery web page
  - reduce or remove between-battery delays in the active polling path
