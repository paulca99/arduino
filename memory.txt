## 2026-05-21

- Current production/master sketch under discussion is `BLE_BMS/BLE_BMS.ino` in `paulca99/arduino`.
- Production stability concern observed on real hardware:
  - the master sketch reportedly ran for about 8 hours and then froze
  - at the same time the inverter reported CAN fail
  - the web page became unresponsive
  - this suggests a whole-device runtime hang/starvation problem rather than a simple UI-only issue
- User clarified that the version currently running in production is generally stable and that web blank/partial loads are relatively rare in normal operation.
- Revised working hypothesis:
  - the worst instability may be caused by disconnected CAN causing repeated transmit retries and repeated `CAN TX FAILED` logging
  - RS485 disconnection/failure may contribute similarly, but likely less severely than CAN
  - serial flood plus retry pressure may starve the loop and make the web page appear flaky or hang
- Desired comms failure behavior agreed for future fixes:
  - CAN:
    - keep the normal healthy CAN transmit cadence when communications are working
    - if CAN transmission fails or the bus is disconnected, do not retry in a tight loop
    - back off to at most one CAN retry every 2 seconds while failures continue
    - throttle failure logging so `CAN TX FAILED` does not flood serial
    - on first success, immediately restore the normal healthy cadence
  - RS485:
    - normal healthy poll interval remains 5 seconds
    - if RS485 polls fail / no valid response is received, do not retry aggressively
    - back off to one RS485 retry every 7 seconds while failures continue
    - throttle repeated failure logging
    - on first valid response, immediately return to the normal 5-second cadence
- A new PR task was started to implement narrow comms-failure hardening only (CAN 2s backoff, RS485 degraded retry, log throttling) rather than a broad web rewrite.
- A separate deep-research task was also started to investigate the long-runtime freeze/hang in `BLE_BMS/BLE_BMS.ino` on master.
- Repository investigation findings from current master `BLE_BMS/BLE_BMS.ino`:
  - RS485 polling runs in a dedicated FreeRTOS task `solisPollTask()`
  - the code currently polls many Solis registers one-by-one using Modbus function 0x04 and `readSolisDocRegU16()`
  - current `SOLIS_REGISTER_SPECS` list contains 21 individual document registers:
    - 33050, 33051, 33052, 33053, 33059, 33072, 33074, 33080, 33081, 33085, 33095, 33129, 33130, 33131, 33132, 33134, 33135, 33136, 33137, 33140, 33142
  - each register is read individually with `count=1`, with `SOLIS_MODBUS_TIMEOUT_MS = 180` and `SOLIS_INTER_REGISTER_DELAY_MS = 15`
  - if the inverter is absent or silent, worst-case one full RS485 poll pass is roughly ~4 seconds of repeated timeout-driven work
  - current RS485 code does not decode Modbus exception responses explicitly; any non-standard response shape is treated as a generic failure
  - current RS485 code currently subtracts 1 from the documented register number before issuing the Modbus read (`docReg - 1`)
- Investigation of current master `CAN_Pylontech.ino`:
  - CAN frames are sent every 100 ms via `sendCANFrames()`
  - each call sends 7 frames
  - `canSend()` currently calls `twai_transmit(..., 10 ms timeout)` for each frame and prints a warning on every failure
  - if CAN is disconnected, this can generate heavy retry pressure and serial flooding, making CAN failure/log starvation one of the strongest freeze suspects
- Solis protocol/spec document identified and used for register audit:
  - user-supplied spec URL: `https://community.openenergymonitor.org/uploads/short-url/kbwIjN8pl8idrCP04UstaxtSmlC.pdf`
  - this document was treated as the authoritative source for resolving current Solis register mysteries
- Key Solis register-map corrections from the spec versus current code:
  - PV/DC input area:
    - spec says 33049 = DC Voltage 1, 33050 = DC Current 1, 33051 = DC Voltage 2, 33052 = DC Current 2, 33053 = DC Voltage 3, 33054 = DC Current 3
    - therefore current code mapping of 33050/33051/33052/33053 as PV1V/PV1I/PV2V/PV2I is shifted/wrong
  - grid/frequency area:
    - spec says 33094 = Grid frequency
    - spec says 33095 = Inverter current status
    - current code treating 33095 as grid frequency is wrong
    - current code treating 33074 as a generic grid voltage is questionable; spec identifies 33074 as BC line voltage
  - battery / meter area:
    - spec says 33129 = Meter current
    - spec says 33130-33131 = Meter active power (S32)
    - spec says 33132 = Storage control switching value
    - spec says 33133 = Battery voltage (0.1 V)
    - spec says 33134 = Battery current (S16, +charge / -discharge)
    - spec says 33135 = Battery current direction (0 charge / 1 discharge)
    - spec says 33136 = LLC bus voltage
    - spec says 33139 = Battery capacity SoC
    - spec says 33140 = Battery health SoH
    - spec says 33141 = Battery voltage from BMS (0.01 V)
    - spec says 33142 = Battery current from BMS (S16, 0.1 A)
    - therefore current code meanings for 33132, 33135, 33136, 33140, and 33142 are wrong
- Practical corrected battery-oriented mapping likely to use in future code:
  - 33133 = battery voltage
  - 33134 = battery current signed
  - 33135 = battery direction
  - 33139 = battery SoC
  - 33140 = battery SoH
  - 33141 = battery voltage from BMS
  - 33142 = battery current from BMS
- Additional register-mystery resolution from the Solis spec:
  - 33059 = reserved
  - 33072 = DC bus half voltage
  - 33129 = meter current
  - 33130-33131 = meter active power (signed 32-bit)
  - 33132 = storage control switching value
  - 33136 = LLC bus voltage
  - 33137 = backup AC voltage phase A
  - 33140 = battery health SoH
- Important possible addressing bug identified from the Solis document:
  - the Solis spec appears to state that 30001-39999 addresses require no extra offset/transform
  - current code does `docReg - 1` before issuing function 0x04 reads
  - if the document is interpreted literally, the current code may be addressing one register early
  - this needs explicit validation against live reads before changing production code blindly
- Strong current interpretation to revisit later:
  - there are two overlapping issues:
    1. current Solis RS485 register map in code is materially wrong in several places
    2. the most likely immediate freeze trigger is still CAN failure/retry/log flooding, with RS485 failure handling as a secondary pressure source
