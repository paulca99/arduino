## 2026-06-09

- Async web library issue resolved for ESP32 Arduino v3:
  - `esp32/charger/pcwifi.cpp` had been using `#include <ESPAsyncWebSrv.h>` / `ESPAsyncWebSrv`, which was the suspicious/incorrect async web library family for the current setup.
  - The working async-web fix was to use the ESP32Async library pair consistently: `AsyncTCP` + `ESPAsyncWebServer` with headers `#include <AsyncTCP.h>` and `#include <ESPAsyncWebServer.h>`.
  - Future ESP32 async web work in this repo should avoid mixing `ESPAsyncWebSrv` / old async-web variants with ESP32Async `ESPAsyncWebServer`.

- Solis ToU/grid-charge behavior learned:
  - Using ToU values `33`/`35` appears to leave grid charging allowed even in `NORMAL` when ToU is stopped.
  - This can explain battery charge power being higher than PV power: the Solis may supplement PV with AC/grid-side power while charging.
  - Earlier interpretation was that `TOU_RUN = 35` means ToU run + allow grid charge, while `TOU_STOP = 33` means ToU stop but grid charge still allowed.
  - If grid charging should only be allowed in emergency/battery-off protection, future code may need a no-grid-charge STOP value (previously discussed candidate: `1`) rather than `33`; verify carefully before changing production behavior.

- Pi/Grafana Solis AC active power work:
  - User wanted to know how much power is coming out of the Solis itself, independent of `solis_grid_power`, because the site grid/meter value can be affected by a separate solar system and the AC-side battery system.
  - The useful live register pair for Solis AC active/inverter power is not the originally decoded parsed keys `33079-33080`; live testing showed that produced zero.
  - With this script's `doc_reg - 1` request convention and parsed-key labeling, the useful pair is parsed keys `33080-33081`, decoded as signed 32-bit watts using high word then low word.
  - Helper used: `s32_from_regs(high, low)` combines `(high << 16) | low` then converts from unsigned 32-bit two's complement to signed.
  - Live Grafana testing confirmed `33080-33081` is a useful AC-side Solis flow metric: it moved with real conditions and reacted during a ~900 W kettle test.

- Observed sign convention for `solis_inverter_active_power` decoded from parsed keys `33080-33081`:
  - positive = AC power coming out of the Solis / Solis supplying AC load or export.
  - negative = AC power going into the Solis / AC-grid-assisted charging side.
  - Therefore do not globally invert this value.
  - Best metric model is to keep the signed raw metric and also emit split metrics:
    - `solis_inverter_active_power` = signed raw AC-side Solis flow
    - `solis_ac_output_power = max(solis_inverter_active_power, 0)`
    - `solis_ac_input_power = max(-solis_inverter_active_power, 0)`
  - Grafana should integrate `solis_ac_output_power` for `Solis AC Output Wh` and `solis_ac_input_power` for `Solis AC Input Wh`.

- Confirmed Solis grid power sign convention:
  - `solis_grid_power < 0` = grid import.
  - `solis_grid_power > 0` = grid export.
  - Therefore:
    - `grid_import_w = max(-solis_grid_power, 0)`
    - `grid_export_w = max(solis_grid_power, 0)`
  - Be careful not to assume the more common opposite sign convention in future dashboards or formulas.

- Example live observations supporting the Solis AC sign convention:
  - Snapshot with PV about `532 W`, battery charging about `703 W`, and Solis AC active power about `-290 W` suggested AC-side input was supplementing PV into battery charge.
  - Later snapshot with PV about `2.02 kW`, battery charging about `2.49 kW`, and signed Solis AC active around `-780 W` again supported negative as AC power going into Solis.
  - During/around the kettle test, signed Solis AC active moved positive when the Solis appeared to be supplying AC-side load, supporting positive as AC output.

- Extra AC-side charger / battery / inverter system discovered:
  - There is another AC-side battery setup connected to the consumer unit, separate from the Solis DC-side batteries.
  - It has an AC charger, batteries, and an inverter. It aims to keep grid power near zero by increasing charge into its own batteries when there is surplus.
  - If grid goes into import, it switches a relay that turns off the charger and turns on the inverter; the inverter then tries to keep grid power near zero by discharging its batteries.
  - This means `solis_grid_power ≈ 0` does not necessarily mean generation equals house load; the AC-side battery system may be absorbing or supplying the difference.
  - This also means previous house-load estimates that only used Solis + fence solar + grid are incomplete unless the AC-side battery system is measured or inferred.

- AC-side JBD Bluetooth BMS monitoring implementation/plan:
  - Better plan than reverse-engineering the blue GTIL/SUN inverter was to monitor the JBD Bluetooth BMS units in the AC-side Solax/Growatt battery setup.
  - Existing ESP32 BLE setup could reliably hold 3 persistent JBD BLE connections but not 4, so AC-side batteries should be handled by a second ESP32.
  - User does not have a second UART/RS485 adapter, so the second ESP32 AC monitor exposes Wi-Fi JSON instead of RS485.
  - New sketch folder: `AC-battery-monitor/`.
  - The sketch should connect once at startup to the two AC-side JBD BLE BMS units, keep connections open, poll packet `0x03` frequently and packet `0x04` less frequently, and expose `/api/bms` over HTTP JSON.
  - Known AC-side BMS MACs to preserve:
    - `AC_Solax`: `a4:c1:37:20:4e:3b`
    - `AC_Growatt`: `a5:c2:37:49:c7:a2`
  - These AC-side BMS readings must be kept completely separate from the DC-side `Growatt2` and `SP14S` batteries that feed Solis/Pylontech CAN control.
  - Do not add the AC-side batteries into the existing `batteries[]` aggregate used for Solis CAN, or Solis CAN current/SoC values could include the wrong battery bank.
  - JBD current sign from the current parser: positive current = battery charging, negative current = battery discharging.
  - AC-side battery DC power is estimated as `voltage * current`:
    - positive = AC-side battery bank is charging
    - negative = AC-side battery bank is discharging
  - This measures DC battery-side power, not exact AC-side charger/inverter power; charger/inverter losses still need efficiency assumptions.
  - The AC monitor JSON aggregate voltage was observed as the sum of the two pack voltages (~93 V) rather than representative pack voltage (~46 V). Until fixed, use per-battery voltages `ac_bms_ac_solax_voltage` and `ac_bms_ac_growatt_voltage`, not aggregate `ac_bms_voltage`, for dashboards.

- ESP32 AC monitor HTTP reliability observation:
  - Python initially showed `Read timed out` while manual `curl` sometimes worked.
  - A likely issue is synchronous `WebServer` only being serviced at the end of a loop dominated by BLE polling/waits.
  - Better pattern for synchronous ESP32 `WebServer` is to call `server.handleClient()` via a helper such as `serviceWeb()` inside BLE wait loops and between batteries, not only once at the end of `loop()`.
  - Python poller should support a configurable HTTP timeout; a longer timeout can be a workaround, but servicing HTTP during BLE waits is the better ESP32-side fix.

- Experimental house-load / AC-bus accounting formulas after adding fence solar and AC-side battery context:
  - Confirmed/current sign assumptions:
    - `solis_inverter_active_power > 0` = Solis AC output
    - `solis_inverter_active_power < 0` = AC going into Solis
    - `solis_grid_power > 0` = export
    - `solis_grid_power < 0` = import
    - `fence_solar_ac_power > 0` = fence inverter AC output
    - `ac_bms_power > 0` = AC-side batteries charging
    - `ac_bms_power < 0` = AC-side batteries discharging
  - Simple signed house-load approximation now used for overview dashboards:
    - `house_load ≈ solis_inverter_active_power + fence_solar_ac_power - solis_grid_power - ac_bms_power`
  - Equivalent split-flow form:
    - `house_load ≈ solis_ac_output_power - solis_ac_input_power + fence_solar_ac_power + ac_bms_discharge_power - ac_bms_charge_power + grid_import_w - grid_export_w`
  - This is still approximate because AC BMS power is battery DC-side power; charger/inverter efficiency losses are not directly measured.
  - For strict AC-bus balance, do not double-count Solis PV/DC battery with Solis AC output. `solis_ac_output_power`/`solis_ac_input_power` are the AC-side Solis flow; Solis PV and DC battery power are useful context but not separate AC-bus sources/drains in the same balance.

- FIT integrity / energy provenance goal:
  - Core motivation for the monitoring: Solis panels are on Feed-in Tariff (FIT), but the fence solar is not.
  - The Solis can pump power into its DC batteries before the FIT/generation meter. If it uses too much fence/AC-side energy to charge those batteries, later discharge through the FIT meter could create a false impression of Solis-generated energy.
  - User wants to ensure that over extended periods the total energy released through the FIT/Solis meter/path is not more than the energy produced by the Solis PV panels.
  - It is acceptable if energy through the FIT path is less than Solis PV production; it is not acceptable to receive FIT payments for energy not produced by Solis panels.
  - Headline FIT check:
    - `Solis AC Output Energy <= Solis PV Generated Energy`
  - Useful FIT dashboard metrics:
    - `Solis PV Generated Wh` = integral of positive `solis_pv_total_power`
    - `Solis AC Output Wh` = integral of `solis_ac_output_power`
    - `FIT Headroom Wh = Solis PV Generated Wh - Solis AC Output Wh`
    - `Potential Non-Solis FIT Wh = max(Solis AC Output Wh - Solis PV Generated Wh, 0)`
    - `FIT Ratio % = Solis AC Output Wh / Solis PV Generated Wh * 100`
  - Interpretation:
    - FIT Headroom >= 0 and FIT Ratio <= 100% is OK.
    - Potential Non-Solis FIT Wh > 0 or FIT Ratio > 100% is a warning/investigation prompt.
  - Important caveat: short ranges can look suspicious if the Solis battery starts full and ends empty, because stored energy may have been generated by Solis PV earlier. FIT integrity is most meaningful over daily/weekly/FIT-review periods and should show Solis battery SoC start/end/delta context.
  - Context panels should include Solis AC Input Wh, Fence Solar Generated Wh, DC battery charge/discharge Wh, AC battery charge/discharge Wh, Solis Battery SoC, Fence Online, and AC Poll OK.
